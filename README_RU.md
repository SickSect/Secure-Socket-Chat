# Secure Chat

TCP-чат со сквозным шифрованием, написанный на Java с нуля. Акцент на понимании
криптографических примитивов, а не на использовании высокоуровневых библиотек.

Текущий релиз (**v0.4**) реализует forward secrecy через эфемерные ECDH-ключи,
персистентные identity-ключи под защитой пароля и аутентифицированный handshake.
Версия **v0.5** (аутентификация, персистентность, жизненный цикл сессий) — в разработке.

> **Примечание:** Этот README — общий обзор. Полные детали протокола, форматы сообщений,
> модель угроз и криптографические константы описаны в документах спецификации в
> [`docs/`](docs/) — [`PROTOCOL_RU.md`](docs/PROTOCOL_RU.md) (русский) и
> [`PROTOCOL_EN.md`](docs/PROTOCOL_EN.md) (английский).

## Возможности

- **Сквозное шифрование (E2E)** — содержимое сообщений шифруется AES-256-GCM ключом
  сессии. Сервер только маршрутизирует шифртекст и не может прочитать сообщения.
- **Forward secrecy** — ключи сессии выводятся из эфемерных ECDH-пар (P-256), которые
  уничтожаются после handshake. Компрометация долгоживущего ключа не раскрывает
  прошлые сессии.
- **Персистентная личность** — у каждого пользователя есть долгоживущая RSA-2048 пара,
  хранящаяся локально в файле под защитой пароля (PBKDF2 + AES-GCM).
- **Аутентифицированный handshake** — эфемерные ключи подписываются долгоживущим
  RSA-ключом; собеседник проверяет подпись, что защищает от подмены ключей
  (man-in-the-middle).
- **Защита от replay-атак** — сообщения несут timestamp и nonce; сервер отклоняет
  устаревшие или повторные сообщения.
- **Транспортный уровень** — весь трафик клиент-сервер шифруется общим ключом (PSK,
  AES-256-GCM), что защищает метаданные от пассивного перехвата.

## Технологии

- **Java 23**
- **Spring Boot** (с v0.5) — REST endpoints, dependency injection, конфигурация
- **PostgreSQL** + **Flyway** — хранение пользователей и миграции схемы
- **Gradle** — сборка
- **TestNG** — тестирование
- Сырые TCP-сокеты для транспорта чата (намеренно без WebSocket/STOMP — чтобы работать
  с протоколом напрямую)

## Быстрый старт

### Требования

- JDK 23
- Docker (рекомендуется) **или** локальный PostgreSQL 16+

### Вариант A — через Docker (рекомендуется)

Поднять PostgreSQL с уже созданной базой:

```bash
docker compose up -d
```

Запускает PostgreSQL на порту `5432` с уже созданной базой `secure_chat`.

### Вариант B — локальный PostgreSQL

Если PostgreSQL уже установлен локально, создайте базу вручную:

```bash
createdb -U postgres secure_chat
```

Задайте учётные данные через переменные окружения (значения по умолчанию):

```bash
export DB_USERNAME=postgres
export DB_PASSWORD=postgres
```

### Запуск сервера

```bash
./gradlew bootRun
```

При старте приложение:
1. Применяет миграции Flyway (создаёт таблицу `users`)
2. Запускает встроенный веб-сервер на порту `8080` (REST endpoints)
3. Запускает TCP-сервер чата на порту `5000`

### Запуск клиента

Клиент — отдельное консольное приложение. Запускайте из IDE
(`org.ugina.client.ChatClient`) — включите **Allow multiple instances** в конфигурации
запуска, чтобы запустить несколько клиентов одновременно.

> Запуск клиента через `./gradlew runClient` **не рекомендуется** — Gradle перехватывает
> stdin и ломает интерактивный ввод. Используйте конфигурацию запуска в IDE.

## Структура проекта

```
src/main/java/org/ugina/
├── Application.java          # Точка входа Spring Boot
├── auth/                     # Абстракция аутентификации (AuthProvider)
│   ├── local/                # Локальная реализация BCrypt + JWT
│   └── exception/            # Исключения аутентификации
├── config/                   # Конфигурация Spring, запуск TCP-сервера
├── controller/               # REST-контроллеры
├── dto/                      # DTO запросов/ответов
├── entity/                   # JPA-сущности
├── repository/               # Репозитории Spring Data
├── client/                   # Клиент чата (консольный TUI)
├── server/                   # Сервер чата (TCP, маршрутизация сообщений)
├── protocol/                 # Типы сообщений, команды, коды ошибок
└── crypto/                   # Криптографические примитивы
    ├── AesCrypto             # AES-256-GCM
    ├── RsaCrypto             # Генерация ключей RSA-2048
    ├── RsaSignature          # Подписи SHA256withRSA
    ├── EcdhCrypto            # Эфемерный ECDH (P-256)
    ├── HkdfCrypto            # HKDF (RFC 5869)
    ├── PbeUtil               # Шифрование на основе пароля (PBKDF2)
    └── keyStorage/           # Файловое хранилище identity-ключей

src/main/resources/
└── db/migration/             # SQL-миграции Flyway
```

## Обзор архитектуры

Система использует четыре типа ключей, у каждого одна ответственность:

| Ключ | Алгоритм | Область | Назначение |
|------|----------|---------|------------|
| Pre-shared key (PSK) | AES-256-GCM | Общий | Шифрует весь трафик клиент-сервер |
| Identity-ключ | RSA-2048 | На пользователя, долгоживущий | Подписывает эфемерные ключи в handshake |
| Эфемерный ключ | ECDH P-256 | На сессию | Выводит общий секрет, затем уничтожается |
| Ключ сессии | AES-256-GCM | На сессию | Шифрует содержимое сообщений (E2E) |

Каждое сообщение шифруется дважды: ключом сессии (сквозное шифрование) и PSK
(транспорт). Сервер видит метаданные, но не содержимое; провайдер сети не видит ни того,
ни другого.

Полная последовательность handshake и форматы сообщений описаны в
[`docs/PROTOCOL_RU.md`](docs/PROTOCOL_RU.md).

## Дорожная карта

### v0.1–v0.3 — готово
- Базовый TCP-чат с несколькими клиентами
- Транспортное шифрование AES-256-GCM
- Сквозное шифрование через гибридную схему RSA + AES
- Защита от replay-атак (timestamp + nonce)

### v0.4 — готово (текущая)
- Forward secrecy через эфемерные ECDH-ключи
- Персистентные RSA identity-ключи под защитой пароля
- Аутентифицированный handshake (эфемерные ключи подписаны identity-ключом)
- Вывод ключей сессии через HKDF
- Архитектурный рефакторинг по слоям

### v0.5 — в разработке
- Регистрация и вход пользователей (REST API)
- Аутентификация на основе JWT (подключаемый `AuthProvider`, готов под Keycloak)
- Персистентность в PostgreSQL с миграциями Flyway
- Жизненный цикл сессии (idle + absolute timeout)
- Rate limiting
- Обнуление памяти для ключей сессии

### v0.6 — планируется
- Per-message forward secrecy (Double Ratchet)
- Коды верификации / safety numbers

### v0.7 — планируется
- Транспорт WebSocket
- Веб-клиент
- Поддержка нескольких устройств

### v1.0 — планируется
- Персистентная история сообщений
- Доставка офлайн-сообщений
- Presence (статус онлайн/офлайн)
