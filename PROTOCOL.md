# Secure Chat Protocol Specification

**Version:** 0.2 (Draft)
**Status:** In development
**Date:** 2026-05-11

---

## 1. Overview

Secure Chat Protocol (SCP) — это прикладной протокол поверх TCP для текстового
обмена сообщениями между несколькими клиентами через центральный сервер.

Версия 0.2 добавляет **шифрование канала** через AES-GCM-256 и унифицирует
формат сообщений в обоих направлениях.

### Возможности v0.2

- Подключение клиента по имени (структурированный handshake)
- Доставка сообщения от одного клиента другому через сервер
- **Шифрование всего трафика AES-256-GCM** с pre-shared key
- **Защита целостности** через GCM auth tag
- Унифицированный JSON-формат во всех направлениях
- Структурированные ответы сервера
- Валидация имени пользователя
- Защита от спуфинга отправителя

### Ограничения v0.2

- **Pre-shared key (PSK):** ключ должен быть заранее доставлен на сервер и
  клиенты любым внеполосным каналом. Эта проблема решается в v0.3 через RSA.
- **Сервер видит сообщения в открытом виде** после расшифровки. Это шифрование
  канала, не end-to-end. E2E будет в v0.3.
- Нет аутентификации клиента: знание PSK = разрешение подключиться.
  Полноценная аутентификация будет в v0.5 через JWT.
- Сообщения не сохраняются: если получатель офлайн, сообщение теряется.
- Нет групповых чатов, только адресные сообщения.

## 2. Transport

| Параметр | Значение |
|----------|----------|
| Транспортный протокол | TCP |
| Порт по умолчанию | 5000 |
| Кодировка | UTF-8 |
| Разделитель сообщений | `\n` (LF) |
| Модель | Клиент-сервер, один сервер на N клиентов |

Каждое сообщение — это **одна строка**, оканчивающаяся `\n`.
В v0.2 содержимое строки — Base64-закодированный шифротекст (см. §4).

---

## 3. Cryptography

### 3.1. Algorithm

- **Cipher:** AES в режиме GCM
- **Key size:** 256 bit (32 байта)
- **IV size:** 96 bit (12 байт) — рекомендованный размер для GCM
- **Auth tag size:** 128 bit (16 байт)
- **Java cipher string:** `AES/GCM/NoPadding`

### 3.2. Key management

В v0.2 используется **pre-shared key (PSK)**:

- Ключ генерируется один раз через утилиту `KeyGenUtil`
- Ключ хранится в `application.properties` (или переменной окружения `SCP_KEY`)
  в формате Base64
- Сервер и каждый клиент должны иметь идентичный ключ

**Ключ не передаётся по сети.** Доставка ключа клиенту — задача администратора
вне рамок этого протокола.

### 3.3. Encryption procedure

Для каждого исходящего сообщения:

1. Сериализовать сообщение в JSON-строку (см. §5)
2. Сгенерировать случайный 12-байтный IV через `SecureRandom`
3. Зашифровать `plaintext_bytes = json.getBytes(UTF_8)` через AES-GCM с PSK и IV
   - На выходе получается `ciphertext || tag` (tag добавляется автоматически)
4. Склеить: `payload = IV || ciphertext || tag` (всего 12 + N + 16 байт)
5. Закодировать в Base64
6. Отправить как строку + `\n`

### 3.4. Decryption procedure

Для каждого входящего сообщения:

1. Прочитать строку до `\n`
2. Декодировать из Base64 → `payload`
3. Разделить: первые 12 байт — IV, остальное — `ciphertext || tag`
4. Расшифровать через AES-GCM с PSK и IV
   - Если auth tag не совпадает — выбросить ошибку, **не доверять данным**
5. Декодировать байты в UTF-8 строку
6. Парсить как JSON-сообщение

### 3.5. Security guarantees

| Угроза | Защита в v0.2 |
|--------|---------------|
| Пассивный перехват трафика | ✅ Шифрование AES-GCM |
| Изменение шифротекста | ✅ GCM auth tag |
| Replay-атака (повтор сообщения) | ❌ В v0.3 добавим timestamp + nonce-tracking |
| MITM с подменой ключа | ⚠️ Защищён только тем, что у атакующего нет PSK |
| Компрометация PSK | ❌ Всё расшифровывается. Решается в v0.3 через эфемерные ключи |
| Спуфинг отправителя | ✅ Сервер подставляет имя из контекста подключения |

## 4. Wire format

```
Base64( IV (12 bytes) || ciphertext (N bytes) || authTag (16 bytes) ) \n
```

После Base64-декодирования и AES-GCM-расшифровки внутри — JSON.

## 5. JSON Message Schema

### 5.1. Client → Server

Объект `ClientMessage`:

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| `commandType` | enum | да | `JOIN`, `SEND_MESSAGE`, `QUIT` |
| `username` | string | для JOIN | Имя пользователя |
| `toClientName` | string | для SEND_MESSAGE | Имя получателя |
| `message` | string | для SEND_MESSAGE | Текст сообщения |

**Изменения относительно v0.1:**
- Добавлен тип `JOIN` — handshake теперь структурированный JSON, не raw string
- Удалено поле `fromClientName` — сервер подставляет имя сам, из контекста подключения

### 5.2. Server → Client

Объект `ServerMessage`:

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| `messageType` | enum | да | `DM`, `DELIVERED`, `ERROR`, `SYSTEM` |
| `fromClientName` | string | для DM | Имя отправителя (если применимо) |
| `text` | string | да | Содержимое для отображения |
| `errorCode` | enum | для ERROR | Код ошибки (см. §7) |

---

## 6. Commands

### 6.1. JOIN

**Направление:** клиент → сервер
**Когда:** первое сообщение после установки TCP-соединения

**ClientMessage:**
```json
{"commandType":"JOIN","username":"Alice"}
```

**Поведение сервера:**

1. Проверить, что `username` валиден (см. §8)
2. Проверить, что имя не занято
3. Зарегистрировать клиента в `ChatRoom` под этим именем
4. Отправить `ServerMessage` с `messageType: SYSTEM` и текстом приветствия

**Возможные ошибки:** `INVALID_NAME`, `NAME_TAKEN`

### 6.2. SEND_MESSAGE

**Направление:** клиент → сервер

**ClientMessage:**
```json
{"commandType":"SEND_MESSAGE","toClientName":"Bob","message":"Hi Bob!"}
```

Заметь: `fromClientName` нет. Сервер берёт его из контекста подключения.

**Поведение сервера:**

1. Найти получателя в реестре
2. Если найден — отправить ему `ServerMessage` с `messageType: DM`,
   `fromClientName` = реальное имя отправителя, `text` = сообщение
3. Отправить отправителю `ServerMessage` с `messageType: DELIVERED`
4. Если не найден — отправить отправителю `ServerMessage` с
   `messageType: ERROR`, `errorCode: RECIPIENT_OFFLINE`

### 6.3. QUIT

**Направление:** клиент → сервер

**ClientMessage:**
```json
{"commandType":"QUIT"}
```

**Поведение сервера:**

1. Удалить клиента из реестра
2. Закрыть соединение

## 7. Error codes

| errorCode | Когда |
|-----------|-------|
| `INVALID_NAME` | Имя не прошло валидацию |
| `NAME_TAKEN` | Имя уже занято подключённым клиентом |
| `RECIPIENT_OFFLINE` | Получатель сообщения не подключён |
| `DECRYPTION_FAILED` | Полученные данные не удалось расшифровать |
| `INVALID_FORMAT` | JSON не парсится или не соответствует схеме |
| `INTERNAL_ERROR` | Ошибка на сервере (логируется) |

---

## 8. Name validation

Допустимое имя пользователя:

- Длина: 1–32 UTF-8 символа
- Допустимые символы: латиница, кириллица, цифры, `_`, `-`
- Regex: `^[\p{L}\p{N}_-]{1,32}$`
- Не может начинаться с `_` или `-`

При невалидном имени сервер отвечает `ERROR` с `errorCode: INVALID_NAME`.

## 9. Connection lifecycle

```
КЛИЕНТ                                        СЕРВЕР
  |                                             |
  |--- TCP connect ---------------------------->|
  |                                             |  ServerSocket.accept()
  |                                             |  → новый ChatHandler в пуле
  |                                             |
  |--- enc(JOIN, "Alice") --------------------->|
  |                                             |  validate, register
  |<-- enc(SYSTEM, "welcome Alice") ------------|
  |                                             |
  |--- enc(SEND_MESSAGE, to:Bob, "hi") -------->|
  |                                             |  lookup recipient
  |<-- enc(DELIVERED, "delivered") -------------|
  |                                             |  send to Bob's handler
  |                                             |
  |--- enc(QUIT) ------------------------------>|
  |                                             |  unregister, close
  |--- TCP FIN -------------------------------->|
```

`enc(...)` обозначает AES-GCM-шифрование сериализованного JSON по §3.

## 10. Backward compatibility

v0.2 **не совместим** с v0.1. Клиент v0.1 не может подключиться к серверу v0.2.

Поскольку проект в стадии разработки, обратная совместимость не поддерживается.

## 12. Known issues and limitations (v0.2)

1. **PSK distribution.** Ключ нужно доставить на каждый клиент вручную.
   Не масштабируется. Решается в v0.3.

2. **No replay protection.** Зашифрованное сообщение можно отправить повторно,
   сервер примет его как новое.

3. **No forward secrecy.** Если PSK скомпрометируют, можно расшифровать
   весь записанный ранее трафик.

4. **No client authentication.** Любой, у кого есть PSK, может подключиться
   под любым свободным именем.

5. **Server sees plaintext.** Сервер расшифровывает все сообщения. Это **не**
   end-to-end. В v0.3 сервер будет передавать зашифрованные данные между
   клиентами без возможности их прочитать.

6. **Single key for all clients.** Все используют один и тот же PSK. Если ключ
   утечёт от одного клиента — все компрометированы.
