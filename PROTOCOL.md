# Secure Chat Protocol Specification

**Version:** 0.1  
**Status:** Draft  
**Date:** 2026-05-11

---

## 1. Overview

Secure Chat Protocol (SCP) — это прикладной протокол поверх TCP для текстового
обмена сообщениями между несколькими клиентами через центральный сервер.

Текущая версия (v0.1) — базовый функциональный протокол **без шифрования и
аутентификации**. Предназначена как фундамент для последующих версий, в которых
будут добавлены защита канала (v0.2), end-to-end шифрование (v0.3) и
веб-транспорт (v0.4+).

### Возможности v0.1

- Подключение клиента по имени (handshake)
- Доставка сообщения от одного клиента другому через сервер
- Корректное отключение по команде клиента

### Ограничения v0.1

- Трафик передаётся **в открытом виде** — любой посредник видит сообщения
- Нет аутентификации — сервер доверяет имени, которое прислал клиент
- Нет защиты от спуфинга — клиент может выдать себя за другого в поле `fromClientName`
- Сообщения **не сохраняются** — если получатель офлайн, сообщение теряется
- Нет групповых чатов и broadcast — только адресные сообщения

## 2. Transport

| Параметр | Значение |
|----------|----------|
| Транспортный протокол | TCP |
| Порт по умолчанию | 5000 |
| Кодировка | UTF-8 |
| Разделитель сообщений | `\n` (LF) |
| Модель | Клиент-сервер, один сервер на N клиентов |

Каждое сообщение в обоих направлениях — это **одна строка**, оканчивающаяся
переводом строки. Парсинг построен на `BufferedReader.readLine()`,
что гарантирует чтение ровно одного сообщения за вызов.

---

## 3. Message Formats

В v0.1 в протоколе используется **два разных формата** сообщений в зависимости
от направления и фазы. Это является техническим долгом, который будет устранён в v0.2.

### 3.1. Client → Server

**Handshake (первое сообщение):** сырая текстовая строка с именем клиента.
Не JSON, без обёртки.

Пример:
```
Alice
```

**Все последующие сообщения:** JSON-объект класса `ClientMessage`, сериализованный
в одну строку.

Пример:
```json
{"toClientName":"Bob","fromClientName":"Alice","message":"hello","commandType":"SEND_MESSAGE"}
```

### 3.2. Server → Client

Все сообщения от сервера к клиенту в v0.1 передаются как **plain text**
(не JSON). Клиент просто отображает строку пользователю.

Примеры:
```
[Alice]: hello
[message delivered]
[server] client:Bob not connected
```

---

## 4. ClientMessage Schema

| Поле | Тип | Обязательное | Описание |
|------|-----|--------------|----------|
| `commandType` | enum | да | Тип команды. Допустимые значения: `SEND_MESSAGE`, `QUIT` |
| `fromClientName` | string | для SEND_MESSAGE | Имя отправителя |
| `toClientName` | string | для SEND_MESSAGE | Имя получателя |
| `message` | string | для SEND_MESSAGE | Текст сообщения |

Для команды `QUIT` все поля кроме `commandType` могут быть `null`.

---

## 5. Commands

### 5.1. SEND_MESSAGE

**Направление:** клиент → сервер

**Описание:** клиент просит сервер доставить сообщение указанному получателю.

**Поля:** все поля `ClientMessage` должны быть заполнены.

**Поведение сервера:**

1. Парсит входящее JSON-сообщение.
2. Проверяет, существует ли получатель (`toClientName`) среди подключённых.
3. Если получатель найден — отправляет ему `[<fromClientName>]: <message>`.
4. После доставки отправляет отправителю `[message delivered]`.
5. Если получатель не найден — отправляет отправителю
   `[server] client:<toClientName> not connected`.

**Известные проблемы:**
- Сервер не проверяет, что `fromClientName` совпадает с зарегистрированным
  именем подключения. Клиент может подделать поле и выдать себя за другого
  пользователя.

### 5.2. QUIT

**Направление:** клиент → сервер

**Описание:** клиент сообщает о намерении отключиться.

**Поля:** только `commandType`.

**Поведение сервера:**

1. Удаляет клиента из реестра комнаты.
2. Завершает обработку соединения.
3. Закрывает сокет.

## 6. Connection Lifecycle

```
КЛИЕНТ                                  СЕРВЕР
  |                                       |
  |--- TCP connect ----------------------->|
  |                                       |  ServerSocket.accept()
  |                                       |  → создаёт ChatHandler
  |                                       |  → отправляет в ExecutorService
  |                                       |
  |--- "Alice\n" (raw name) ------------->|
  |                                       |  ChatRoom.joinClient(name)
  |                                       |
  |--- {"commandType":"SEND_MESSAGE",...} ->|
  |                                       |  ChatRoom.sendMessage(msg)
  |<-- "[Alice]: hello\n" ----------------|  (получателю)
  |<-- "[message delivered]\n" -----------|  (отправителю)
  |                                       |
  |--- {"commandType":"QUIT"} ------------>|
  |                                       |  ChatRoom.leaveClient(name)
  |--- TCP close ------------------------->|  socket.close()
```

## 7. Error Handling

Обработка ошибок в v0.1 минимальна и носит характер «лучшее усилие».

| Сценарий | Текущее поведение | Желаемое поведение (v0.2) |
|----------|-------------------|---------------------------|
| Имя занято при JOIN | Сервер логирует, клиенту ничего не сообщает | Отправить ошибку, клиент должен ввести новое имя |
| Невалидный JSON | Сервер ловит Exception и закрывает соединение | Отправить структурированный ответ об ошибке |
| Получатель не подключён | Текстовое сообщение отправителю | Структурированный ответ с кодом ошибки |
| Спуфинг `fromClientName` | Сервер принимает любое значение | Сервер подставляет имя из контекста подключения |
| Клиент аварийно отключился | Сервер ловит IOException, удаляет из реестра | Без изменений |

## 8. Versioning

Текущая версия — **0.1**. Версия в самом протоколе пока не передаётся.
Начиная с v0.2, в handshake появится поле версии.

### Запланированные версии

| Версия | Изменения |
|--------|-----------|
| **v0.2** | Унификация JSON во всех направлениях. Защита канала через AES (single key, выдаваемый сервером). Имя пользователя как часть структурированного JOIN-сообщения. Структурированные ответы об ошибках. |
| **v0.3** | RSA-обмен ключами между клиентами. End-to-end шифрование сообщений: сервер передаёт зашифрованные данные, не имея возможности их прочитать. |
| **v0.4** | Переход с raw TCP на WebSocket. Поддержка веб-клиента в браузере. |
| **v0.5** | Аутентификация через JWT. Регистрация пользователей. Постоянное хранилище. |
| **v1.0** | История сообщений, доставка офлайн, presence-статусы. |

## 9. Known Issues and Limitations

Это **честный список** известных недостатков текущей реализации.
Они не являются багами — это **спроектированные** ограничения v0.1.

1. **Plain text traffic.** Любой посредник между клиентом и сервером может
   читать всё содержимое сообщений.

2. **No authentication.** Сервер доверяет имени, которое прислал клиент.
   Любой может представиться любым именем, если оно ещё не занято.

3. **Spoofing of sender.** Поле `fromClientName` в SEND_MESSAGE не проверяется
   на соответствие реальному подключению. Клиент A может отправить сообщение
   с `fromClientName: "B"`, и сервер его обработает.

4. **No replay protection.** Записанное сообщение можно отправить повторно,
   сервер не отличит копию от оригинала.

5. **Format inconsistency.** Server→Client сообщения — plain text, Client→Server —
   JSON (после handshake). Это усложняет реализацию структурированной обработки
   ошибок на клиенте.

6. **Asymmetric handshake.** Первое сообщение от клиента — сырая строка с именем,
   а не JSON. Это исключение из общего правила протокола.

7. **No name validation.** Сервер принимает любую строку как имя, включая пустую,
   с пробелами, с управляющими символами, или произвольной длины.

8. **Lost messages.** Если получатель не подключён в момент отправки, сообщение
   теряется навсегда.

9. **No graceful shutdown.** При отключении сервера подключённые клиенты получат
   IOException без какого-либо уведомления о причине.

## Appendix A: Full message examples

### Successful flow

**Connection:**
```
TCP SYN → SYN-ACK → ACK
```

**Handshake:**
```
Client → Server:    Alice
```

**Message exchange (Alice → Bob):**
```
Client → Server:    {"commandType":"SEND_MESSAGE","fromClientName":"Alice","toClientName":"Bob","message":"Hi Bob!"}
Server → Bob:       [Alice]: Hi Bob!
Server → Alice:     [message delivered]
```

**Recipient offline:**
```
Client → Server:    {"commandType":"SEND_MESSAGE","fromClientName":"Alice","toClientName":"Charlie","message":"hello"}
Server → Alice:     [server] client:Charlie not connected
```

**Disconnect:**
```
Client → Server:    {"commandType":"QUIT"}
TCP FIN → FIN-ACK → ACK
```
